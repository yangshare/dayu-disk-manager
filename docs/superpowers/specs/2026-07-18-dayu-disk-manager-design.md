# dayu-disk-manager 设计规格

> 状态：已批准　|　日期：2026-07-18　|　阶段：头脑风暴完成，待编写实现计划

## 1. 项目定位与核心价值

**dayu-disk-manager** 是一款 Windows 桌面磁盘治理工具，核心功能是通过 **junction（目录联接）** 把 C 盘的大体积目录迁移到其他盘，并在原位建立链接，从而释放 C 盘空间。项目名取"大禹治水"之意——疏导归流，而非东搬西塞。

**定位：** 面向懂一点电脑的用户。有图形界面但保留专业控制力，介于傻瓜式"C 盘搬家"和极客命令行工具之间。预设常见场景 + 支持自定义目录，默认安全（可预览、可还原）。

**三大核心价值：**

1. **智能识别** — 扫描 C 盘自动发现并标注可迁移的大目录（微信、Steam、开发工具缓存等），扫描结果带"一键迁移"。
2. **安全迁移** — 复制→校验→建链→删原（走回收站）三步法 + 可靠回滚 + 完整操作日志。
3. **统一治理** — 所有迁移点归集到统一仓库 `D:\Migrated`（可配置），结构清晰、历史可查。

**技术栈：** Tauri 2 + Rust + Vue 3。junction 无需管理员权限，整个工具可低权限运行。

**技术策略约束：** 只处理目录，只用 junction，不支持单文件链接（单文件迁移对释放 C 盘空间无实际意义）。

## 2. 核心功能模块

软件由 5 个功能模块组成：

### 2.1 扫描分析（Scan）
- 扫描 C 盘目录树，计算各目录占用大小。
- 自动识别预设场景（通讯办公 / 游戏平台 / 开发工具）的安装位置，标注"这是什么、占多大、建议迁移"。
- 未命中预设的大目录（按体积阈值，默认 >500MB）列出，提供"自定义迁移"入口。
- 扫描结果可排序、可筛选，是发现"该迁什么"的入口。

### 2.2 迁移（Migrate）
- 执行核心三步法：复制到统一仓库 → 校验完整性 → 建 junction → 原目录移入回收站。
- 迁移前预检：目标盘剩余空间是否足够、源目录是否被进程占用、是否为系统关键路径（拒绝迁移）。
- 操作前展示"将要做什么"的清单，用户确认后执行。
- 支持取消（中途取消时安全回滚到迁移前状态）。
- 迁移进度可视化（进度条 + 当前阶段）。

### 2.3 软链接管理（Links）
- 列出所有由本工具创建的 junction（及系统已有的 junction，标注来源）。
- 每条链接可操作：还原（删 junction、把数据搬回原位）、删除链接（保留迁移后数据，仅断开链接）、打开原位置。
- 检测失效链接（目标被手动删除的），提供清理。

### 2.4 操作历史（History）
- 完整记录每次迁移、还原、删除操作（时间、源路径、目标路径、结果）。
- 可按操作类型/时间筛选查看。
- 双向可追溯：从历史能跳到对应软链接，从软链接能看迁移历史。

### 2.5 设置（Settings）
- 统一迁移仓库路径（默认 `D:\Migrated`，可改到其他盘）。
- 扫描偏好（大目录阈值、要排除的路径）。
- 数据存储位置查看、导出操作日志。

## 3. 架构与模块边界

按 Tauri 标准分层：

```
┌─────────────────────────────────────────────────┐
│  Frontend (Vue 3 + TypeScript)                  │
│  ├─ ScanView      扫描结果展示与交互              │
│  ├─ MigrateView   迁移预检清单与进度              │
│  ├─ LinksView     软链接列表与管理操作            │
│  ├─ HistoryView   操作历史                       │
│  └─ SettingsView  设置                          │
├──── Tauri IPC (invoke / event) ────────────────┤
│  Backend (Rust)                                 │
│  ├─ scanner        扫描 + 预设场景识别            │
│  ├─ migrator       三步法状态机 + 回滚            │
│  ├─ junction      junction 创建/删除/校验         │
│  ├─ safety        预检（空间/占用/路径黑名单）     │
│  ├─ history       操作日志读写                   │
│  ├─ store         迁移记录与配置（本地数据）        │
│  └─ win32         Win32 API 封装                 │
└─────────────────────────────────────────────────┘
```

**各单元职责与接口：**

| 单元 | 职责 | 对外接口（给其他单元） | 依赖 |
|------|------|----------------------|------|
| `scanner` | 遍历目录算体积、匹配预设场景 | `scan(root) -> Vec<ScanItem>` | `store`（读预设场景配置） |
| `safety` | 迁移前预检 | `precheck(src, dst) -> Report` | `win32`（查盘空间/占用进程） |
| `migrator` | 执行三步法 + 进度事件 | `migrate(src, dst, on_progress)` | `safety`、`junction`、`history`、`store` |
| `junction` | junction 的创建/删除/读取/校验 | `create`、`remove`、`resolve`、`verify` | `win32` |
| `history` | 操作日志追加与查询 | `append(op)`、`list(filter)` | `store` |
| `store` | 配置与迁移记录的持久化 | 读写 JSON | 无 |
| `win32` | Win32 API 封装（盘空间、junction API、文件占用） | 薄封装函数 | 无 |

**关键边界原则：**
- `migrator` 是状态机的核心，但它**只编排**——实际建链委托 `junction`，预检委托 `safety`，记录委托 `history`。这样三步法的逻辑可单独测试，不耦合 Win32 细节。
- `win32` 是唯一的平台边界，其他单元不直接碰系统 API。未来若考虑跨平台，只改这一层。
- `scanner` 和 `migrator` 通过 Tauri event 向前端推进度，前端不轮询。

## 4. 核心数据流 — 一次完整迁移

以"迁移微信文件"为例，走通主流程，验证各模块协作：

```
1. 扫描阶段
   用户点"扫描 C 盘"
   → scanner.scan("C:\")
   → 遍历 + 匹配预设场景，识别到 C:\Users\xxx\Documents\WeChat Files
   → 前端展示：「微信文件，32.4 GB，建议迁移」+ [迁移] 按钮

2. 预检阶段
   用户点[迁移]
   → store 读配置：仓库 = D:\Migrated
   → migrator 调 safety.precheck(src, "D:\Migrated\WeChat Files")
     - 查 D 盘剩余空间 ≥ 32.4G？✓
     - 源目录被进程占用？检测 WeChat.exe 是否运行（占用则提示先关闭）
     - 路径在黑名单？（系统关键目录拒绝）✓
   → 前端展示预检清单，用户[确认]

3. 迁移阶段（三步法状态机）
   阶段a 复制：copy C:\...\WeChat Files → D:\Migrated\WeChat Files
     - 进度事件 on_progress("copying", 45%)
     - 若失败/取消 → 清理目标残留，回滚到迁移前
   阶段b 校验：对比源与目标的文件数、总字节数
     - 不一致 → 中止，保留目标供排查，标记"待人工确认"
   阶段c 建链：
     - 将源目录改名 → C:\...\WeChat Files.old（暂存，非直接删）
     - junction.create("C:\...\WeChat Files", "D:\Migrated\WeChat Files")
     - 校验 junction 解析正常
   阶段d 删原（走回收站）：
     - 将 .old 目录移入回收站（SHFileOperation + FOF_ALLOWUNDO）
     - 记录：源 .old 已在回收站，可彻底回收

4. 记录阶段
   → history.append({op: "migrate", src, dst, time, result: "ok"})
   → store 写迁移映射：{src, dst, .old回收站状态, time}
   → 前端跳转到 LinksView，新链接高亮

5. 回滚（用户事后在 LinksView 点[还原]）
   → junction.verify 先确认链接仍指向有效目标
   → 删 junction → 把 D:\Migrated\WeChat Files 复制回 C:\...\WeChat Files
   → 校验 → history.append({op: "restore", ...})
   → 清理 D:\Migrated\WeChat Files（移回收站）
```

**几个设计要点：**

- **`.old` 暂存 + 回收站而非直接删原**：建链成功前原目录绝不删；建链成功后原目录走回收站，给用户最后一道后悔药。这是"中等安全"方案的关键。
- **校验用文件数+字节数对比**，不用逐字节 hash（太慢，30G 目录会算很久）。字节数+文件数一致已足够可靠，且是迁移工具行业惯例。
- **进度用 Tauri event 推送**，前端被动接收，不轮询。
- **进程占用检测**：迁移前若源目录被占用，提示用户先关应用；不强杀进程（危险且不专业）。

## 5. 错误处理与边界情况

磁盘工具最容易翻车的是异常场景。明确处理策略：

**迁移阶段的失败/中断处理：**

| 阶段 | 异常 | 处理 |
|------|------|------|
| 复制中 | 磁盘满 / 权限 / 中断 | 清理目标盘残留 → 状态恢复迁移前，标记失败 |
| 复制中 | 用户点取消 | 同上，安全回滚，记录"用户取消" |
| 校验 | 文件数/字节数不一致 | 保留目标盘完整副本（不删），标记"待人工确认"，提示用户排查 |
| 建链 | 源目录改名（→ .old）失败 | 通常因目录被占用。中止迁移，目标副本保留，提示用户关闭占用进程后重试；不强制改名 |
| 建链 | junction 创建失败 | 已复制的目标保留，.old（若已改名）还原回原名，标记失败 |
| 删原 | 移回收站失败 | junction 已建好、数据已迁移成功，仅删除原目录失败——提示用户可手动删 .old（安全，数据已在新位置） |

**关键原则：失败时永远优先保数据，宁可留垃圾也不删数据。** 每个阶段的失败都有明确状态，前端可看到"卡在哪一步"，并提供"清理残留"入口。

**其他边界情况：**

- **源目录被占用**：预检阶段检测占用进程，提示"请先关闭 XXX"。提供"我已关闭，重试"按钮。
- **目标盘空间不足**：预检直接拦截，建议换盘或清理目标盘。
- **系统关键路径**：硬编码黑名单（`C:\Windows`、`C:\Program Files` 下的系统组件、`System32` 等），扫描和迁移都拒绝，给出"系统关键目录，不建议迁移"提示。
- **重复迁移**：扫描时若发现某目录已经是 junction（已被本工具或他人迁移过），标注"已迁移"，不重复操作。
- **目标已存在**：`D:\Migrated\WeChat Files` 若已存在（重复迁移/用户旧数据），预检提示冲突，要求用户确认覆盖或改名。
- **junction 失效**：LinksView 检测到链接指向的目标不存在，标注"失效"，提供"清理失效链接"。
- **跨盘符根目录**：仓库不能选 C 盘本身（会形成无意义循环迁移），设置里校验。
- **权限**：junction 不需管理员权限，正常情况工具直接运行。极少数受保护目录（如其他用户的 `Documents`）建链失败时，提示需要权限并跳过，不强行提权。

## 6. 数据持久化与配置

**1. 存储位置**

所有数据放在 `%APPDATA%\dayu-disk-manager\`（即 `C:\Users\xxx\AppData\Roaming\dayu-disk-manager\`）：

```
%APPDATA%\dayu-disk-manager\
├─ config.json          配置（仓库路径、扫描阈值、黑名单等）
├─ migrations.json      迁移映射记录（src ↔ dst ↔ 回收站状态）
└─ history.jsonl        操作历史（追加写，每行一条 JSON）
```

**2. 数据结构**

`config.json`：
```json
{
  "repository": "D:\\Migrated",
  "scan": {
    "minSizeMB": 500,
    "excludePaths": ["C:\\Windows", "C:\\Program Files\\WindowsApps"]
  },
  "presets": [ ]
}
```

`migrations.json`（迁移映射，回滚的依据）：
```json
[
  {
    "id": "uuid",
    "source": "C:\\Users\\xxx\\Documents\\WeChat Files",
    "target": "D:\\Migrated\\WeChat Files",
    "preset": "wechat",
    "createdAt": "2026-07-18T10:30:00Z",
    "status": "active",
    "recycleBinRef": ""
  }
]
```

`history.jsonl`（操作流水，每行一条）：
```json
{"op":"migrate","id":"uuid","src":"...","dst":"...","result":"ok","time":"...","durationSec":120}
```

**3. 设计要点：**

- **migrations.json vs history.jsonl 分工**：前者是"当前状态"（哪些链接还活着，回滚查它），后者是"流水账"（审计查它）。前者读多写少、整体改写；后者只追加，永不改历史记录。
- **预设场景可扩展**：`presets` 写在 config 里，内置默认值首次启动注入。未来加新场景（如 Epic 游戏）只改预设，不动代码。
- **数据安全**：migrations.json 是回滚的命根子，每次写入前先备份为 `.bak`，写失败回滚。绝不允许该文件损坏后用户无法还原。
- **配置校验**：仓库路径启动时校验——不能是 C 盘、不能是网络路径、所在盘需可写。

## 7. 测试策略

磁盘工具测试难度高——动真实文件系统。策略是**分层测试，核心逻辑脱离真实磁盘**：

**1. 单元测试（Rust，`cargo test`，不依赖真实磁盘）**

| 单元 | 测试要点 | 手段 |
|------|---------|------|
| `scanner` | 体积计算、预设场景路径匹配 | 用 `tempfile` 建临时目录树，构造 mock 文件结构 |
| `safety` | 空间判断、黑名单匹配、占用检测逻辑 | 路径黑名单用纯数据测试，不碰真实进程 |
| `migrator` | 三步法状态机：成功/各阶段失败/取消的回滚 | **把文件操作抽象成 trait**，测试用 mock 实现，只验编排逻辑 |
| `junction` | 创建/删除/校验逻辑 | 用 `tempfile` 真实建 junction（junction 不需管理员权限，CI 可跑） |
| `store` | 配置读写、备份回滚 | 用临时目录当数据目录 |

**关键设计：`migrator` 依赖一个 `FileOps` trait**（`copy`、`remove`、`to_recyclebin`、`create_junction`），生产用真实实现，测试用 mock。这样三步法逻辑的测试不碰真实文件系统，能在毫秒级跑完所有分支。

**2. 集成测试（Rust，少量，真实文件系统）**

- 用 `tempfile` 建一个完整的 mock C 盘结构（微信目录、Steam 目录、随机大目录），跑通"扫描→预检→迁移→建链→记录"全链路。
- 验证迁移后 junction 可被 Windows 正常解析（用 `std::fs::read_link` 或 Win32 `GetFinalPathNameByHandle`）。
- 验证回滚后数据完整。

**3. 手工/端到端验证清单（不在 CI，发布前跑）**

真实环境上验证的关键场景：
- 微信/Steam 真实迁移（关闭应用后），迁移后应用能正常启动、找到文件。
- 迁移中拔网线/断电模拟（杀进程）→ 重启工具，验证残留清理与状态恢复。
- 回收站还原验证（`.old` 从回收站恢复）。
- 失效链接清理。

**4. 不做的测试**

- 不做 UI 自动化测试（Vue 组件用 Vitest 单测关键逻辑即可，E2E 投入产出比低）。
- 不做性能基准测试（首版不需要）。

## 8. 第一版范围（YAGNI 边界）

**包含：**
- 预设场景：通讯办公（微信/QQ/钉钉/企业微信）、游戏平台（Steam/Epic/WeGame）、开发工具（IDE 缓存/npm/Maven/Gradle/Docker/pip）、自定义目录。
- 扫描 + 智能识别 + 迁移三步法 + 链接管理 + 历史记录 + 设置。

**明确不包含（留到后续版本）：**
- 浏览器缓存迁移（缓存会自重建，部分浏览器不支持软链，收益低）。
- 系统级用户文件夹迁移（文档/下载/桌面，Windows 原生支持但有兼容性风险，需单独严谨设计）。
- 单文件链接。
- 跨平台（仅 Windows）。
- 性能基准、UI E2E 自动化测试。
